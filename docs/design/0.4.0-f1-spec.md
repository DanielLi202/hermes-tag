# F-1 Implementation Spec — `im:message.group_msg` capability check + explicit diagnostic

Status: spec for implementation. 2026-07-09. Author: Claude (architect). Implementer: Codex.
Parent plan: [0.4.0-plan.md](0.4.0-plan.md) §F-1. Sibling precedent: [0.4.0-f2-spec.md](0.4.0-f2-spec.md) (PR #13).

## Goal

Tier-0 full-group ingest depends on the sensitive Feishu scope `im:message.group_msg`, but the
plugin only reads the **config claim** (`granted_scopes:` in config → `TagConfig.has_group_msg_scope`).
When the config claims the scope but the Feishu app does not actually hold it, Feishu silently never
delivers non-@ group messages: Tier-0 stays thin, answers degrade, and the user blames the plugin
("silent no-response" — the top support-pain class in both competitors). Verify the claim against
reality once per process and tell the admin loudly and precisely when it is wrong.

**Verified facts (live-probed on the prod app 2026-07-09 — trust these, do not re-verify):**

- `GET /open-apis/application/v6/scopes` works with a plain `tenant_access_token` — **no extra
  permission needed** (returned code 0 with 37 scopes on the prod app). Response items:
  `{scope_name: str, grant_status: int, scope_type: str}`; granted ⇔ `grant_status == 1`;
  a scope **absent from the list** counts as not granted.
- The installed lark SDK has this API: `client.application.v6.scope.list(ListScopeRequest)` (and
  `alist`), models in `lark_oapi.api.application.v6` (`ListScopeRequest`, `ListScopeResponse`).
- The base hermes-agent Feishu adapter already has a best-effort probe precedent (bot-name
  hydration): it calls `await self._run_blocking(self._client.application.v6.application.get, req)`
  and logs at debug/warning on failure, never raises. Mirror that invocation style.
- The base adapter's `send(chat_id=...)` picks `receive_id_type` by prefix — an `ou_…` id sends a
  **DM** to that user (verified in base adapter source). So DM-ing admins is just
  `self.send(admin_open_id, text)`.

## Design (decided — do not redesign)

- **Mechanism (Layer B, Feishu only)**: new seam method fetching live scopes. Safe default =
  `None` ("unknown"), so Slack/DingTalk and stub mode are zero-touch.
- **Decision (Layer A)**: compare live scopes against the config claim once per process, lazily on
  the first inbound event in an enabled chat; on mismatch → `logger.warning` + audit row + one-time
  DM to each configured admin; surface the cached verdict in `/tag status`. Unknown → stay silent
  (debug log only). The check must NEVER delay-fail or break message handling.
- **No behavior change to ingest**: this release only *reports*; it does not flip `receive_all`.

## Exact changes

### 1. `src/hermes_tag/base.py` (Layer A)

- Module-level logger (match `__init__.py` style): `logger = logging.getLogger(__name__)`.
- `TagAdapterMixin.__init__`: add `self.capability_check: dict[str, Any] = {"status": "pending"}`
  and `self._capability_check_started = False`.
- New seam default (below `normalize_inbound_identity`):
  ```python
  async def probe_granted_scopes(self) -> dict[str, bool] | None:
      return None
  ```
- New Layer A orchestration method `_maybe_verify_capabilities(self, chat_id: str) -> None`
  (async). Behavior:
  1. If `self._capability_check_started` → return. Set the flag BEFORE any await (single event
     loop ⇒ no double-probe).
  2. `scopes = await asyncio.wait_for(self.probe_granted_scopes(), timeout=5)` inside
     `try/except Exception` — any exception or `None` result →
     `self.capability_check = {"status": "unknown"}`, `logger.debug(...)`, return.
  3. Let `claimed = self.tag.has_group_msg_scope` and
     `live = bool(scopes.get("im:message.group_msg"))` (constant
     `REQUIRED_GROUP_SCOPE = "im:message.group_msg"` at module level, next to the other module
     constants).
  4. Verdicts:
     - `claimed and not live` → `status="mismatch"`: `logger.warning`, audit
       `self.store.audit("capability_mismatch", chat_id, <json: scope, claimed, live>)`, and DM
       each entry of `self.tag.admins` with `CAPABILITY_MISMATCH_NOTICE["zh"]` via
       `await self.send(admin_id, notice)` — each DM in its own `try/except` → `logger.debug` on
       failure. DMs happen at most once per process (the started-flag already guarantees this).
     - `not claimed and live` → `status="upgrade_available"`: same channel/audit
       (`capability_upgrade` audit type) with `CAPABILITY_UPGRADE_NOTICE["zh"]`.
     - otherwise → `status="ok"`, no sends, no audit.
     Always set `self.capability_check = {"status": ..., "scope": REQUIRED_GROUP_SCOPE,
     "claimed": claimed, "live": live}` (omit scope/claimed/live for unknown/pending).
  5. The whole method body is wrapped so no exception can propagate (outermost
     `try/except Exception: logger.debug(..., exc_info=True)`).
- Call site — `_dispatch_inbound_event`, AFTER the enabled-chats early return, BEFORE
  `engine.handle_message`:
  ```python
  await self._maybe_verify_capabilities(_chat_id(event))
  ```
  (Feishu is the only platform routing through here, and the only one with a real probe; Slack/
  DingTalk overrides of `handle_message` are intentionally untouched.)
- `preflight_status()`: add top-level key `"capability_check": dict(self.capability_check)`.

### 2. `src/hermes_tag/i18n.py`

Two new dicts following the `ENABLE_NOTICE` pattern exactly (zh + en keys; zh is what gets sent):

- `CAPABILITY_MISMATCH_NOTICE` — must state: config claims `im:message.group_msg` but the Feishu
  app does not actually hold it; consequence (non-@ group messages are never delivered, so the
  bot's group context stays @-only); action (grant the scope in 开发者后台「权限管理」and republish
  the app version, or remove it from `granted_scopes` in the plugin config to silence this).
- `CAPABILITY_UPGRADE_NOTICE` — the app holds `im:message.group_msg` but the config does not claim
  it; action (add it to `granted_scopes` in the plugin config to enable full group context).

Keep each under ~3 sentences; same register as `ENABLE_NOTICE`.

### 3. `src/hermes_tag/platforms/feishu.py` (Layer B)

- `FeishuTagAdapter.probe_granted_scopes(self) -> dict[str, bool] | None`:
  ```python
  async def probe_granted_scopes(self):
      client = getattr(self, "_client", None)
      run_blocking = getattr(self, "_run_blocking", None)
      if client is None or run_blocking is None:
          return None  # stub mode / degraded base
      try:
          from lark_oapi.api.application.v6 import ListScopeRequest
          response = await run_blocking(client.application.v6.scope.list, ListScopeRequest.builder().build())
          return _scopes_from_response(response)
      except Exception:
          return None
  ```
- Module-level pure helper (unit-testable without a client):
  ```python
  def _scopes_from_response(response: Any) -> dict[str, bool] | None:
  ```
  Returns `None` if `response` is falsy or `not response.success()` (use
  `getattr(response, "success", None)` defensively); otherwise
  `{s.scope_name: getattr(s, "grant_status", None) == 1 for s in response.data.scopes}` with the
  same defensive getattr chain (`data`, `scopes` may be missing → `None`). Add
  `_scopes_from_response` to `__all__` (keep the list sorted).

### 4. Tests — new file `tests/test_capability_check.py`

Reuse the fixture idiom from `tests/test_identity_normalization.py` (FakeAdapter on
`TagAdapterMixin` + stubs env var; note the `setattr(FakeAdapter, "_fetch_" + "reply" +
"_media_refs", ...)` idiom — required by the forbidden-substring guard in `test_foundation.py`,
do not inline that method name as a literal). MUST cover:

1. **Default unknown**: adapter without a probe override → after first dispatched event,
   `capability_check == {"status": "unknown"}`, zero outbound sends, no `capability_mismatch`
   audit rows.
2. **Mismatch → one-time admin DM**: probe override returns `{"im:message.group_msg": False}`
   with config claiming the scope and `admins=("ou_admin1",)` → after first event: warning-path
   taken, exactly one DM to `ou_admin1` containing the mismatch notice, one `capability_mismatch`
   audit row, `capability_check["status"] == "mismatch"`; after a SECOND event, still exactly one
   DM and one probe call (count probe invocations).
3. **OK**: probe returns `{"im:message.group_msg": True}`, config claims → `status == "ok"`,
   zero notices.
4. **Upgrade available**: config `granted_scopes=[]`, probe returns True for the scope →
   `status == "upgrade_available"`, one DM with the upgrade notice, and `receive_all` stays
   `False` (no ingest behavior change).
5. **Probe failure is harmless**: probe raises → event still dispatched/handled normally,
   `status == "unknown"`.
6. **`_scopes_from_response` parsing**: SimpleNamespace fakes — success shape with grant_status
   1/0 mix; `success()` False → None; missing `data.scopes` → None.
7. **Status surfacing**: `preflight_status()["capability_check"]` reflects the cached verdict.
