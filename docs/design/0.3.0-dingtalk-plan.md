## 0. 核心结论（决定整个计划形态）

**Hermes v0.17.0 已经自带一个基础钉钉适配器**：`plugins/platforms/dingtalk/adapter.py`
（`DingTalkAdapter(BasePlatformAdapter)`，~71KB），作为 bundled 插件以 `name="dingtalk"` 注册，
基于 `dingtalk-stream` SDK 的 Stream 模式，已支持文本/图片/语音/视频/富文本/文件、群 @ 门控、
free-response 群、按用户白名单。

因此，**为 hermes-tag 增加钉钉支持 = 复刻 Slack 的接入模式**（在已有 base adapter 上加一层薄 mixin），
**而不是从零写一个 base adapter**。这把 0.3.0 从"几千行平台底层 + 上游 PR"降级为"一个 `platforms/dingtalk.py`
文件 + 测试 + 文档"，且 **Layer A（`core.py`/`context.py`/`base.py`/`i18n.py`）一行不动**（满足 roadmap 的 I7：
不改 `PlatformSeam` 签名，不加新依赖到核心）。

这与此前 Slack 的接入路径同构（参照 `src/hermes_tag/platforms/slack.py`）；钉钉是第三个渠道。

## 1. 架构与覆盖（override）机制（已验证）

- 平台注册表 `gateway/platform_registry.py` 的 `register()` 是 **last-writer-wins**：
  同名再注册即替换（"this lets plugins override built-in adapters if desired"）。
- 加载顺序：bundled 插件（`plugins/platforms/dingtalk/`）先加载，pip entry-point 插件（hermes-tag，
  经 `hermes_agent.plugins`）后加载 → hermes-tag 的 `name="dingtalk"` 注册**覆盖** bundled 的那一个。
  这正是 feishu/slack 今天能被覆盖的同一机制（已在生产 live-verified）。
- 前提：目标机上 bundled 钉钉插件存在/启用，且 `dingtalk-stream` 已装（机器上已确认安装）。

**入站拦截点（关键差异）：钉钉用 `handle_message(event)`，与 Slack 相同，而非 Feishu 的 `_dispatch_inbound_event`。**
钉钉 `_on_message` 自己构造 `MessageEvent` 后调用 `self.handle_message(event)`（继承自
`BasePlatformAdapter.handle_message`，base.py:4229）。所以 `DingTalkTagAdapter` 应像 `SlackTagAdapter`
一样**覆盖 `handle_message`**，并在 `chat_id not in enabled_chats` 时 `return await super().handle_message(event)`。

## 2. Seam 逐项映射（钉钉具体实现）

| PlatformSeam | 钉钉实现 | 说明 |
|---|---|---|
| 入站拦截 | override `handle_message(event)` | enabled_chats 命中才进 engine，否则 `super().handle_message(event)`；复刻 SlackTagAdapter（slack.py:140-145）。钉钉无原生 slash command，不需要 Slack 那个 `_is_native_slack_command` 中间分支 |
| 模型分发 | override `dispatch_to_model(event)` → `await super().handle_message(event)` | **必需，易漏。** engine 在 enhance 后调用 `seam.dispatch_to_model`（core.py:361）；`TagAdapterMixin` 默认实现走 Feishu 的 `super()._dispatch_inbound_event`（base.py:177-178），而钉钉 base adapter **没有该方法** → 不覆盖会 AttributeError。必须像 Slack 一样覆盖为 `super().handle_message`（slack.py:147-148） |
| `platform_name` | `"dingtalk"` | |
| `receive_all` | `True`（群内实际收不到非 @ 消息，见 §3.1/§4.4） | 钉钉只下发「@ 机器人」群消息 + 单聊 → 群里 Tier-0 拿不到环境消息；`receive_all=True` 仅在单聊有意义。保留 `True` 与 Slack 一致、无害 |
| `is_mentioned` | DM(`source.chat_type=="dm"`)→True；群→`raw_message.is_in_at_list` | 钉钉用结构化 `is_in_at_list`，非文本解析。**采用 Feishu 模型：命令也需 @ 或 DM**——不照搬 Slack 的 `/tag` 文本快捷判定（那是为 Slack 原生 slash command 设的，slack.py:150-158）；钉钉 `/tag` 只是普通文本，与 §8「非 @ 的 /tag 无回复」保持一致 |
| `_fetch_reply_media_refs` | stub → `[]` | 钉钉 stream 无"取父消息"能力，事件也无 reply_to |
| `_download_media` | stub → `("","")` | 同上，复刻 Slack |
| `send` / `send_to_platform` | 继承 `DingTalkAdapter.send(chat_id, content, reply_to=, metadata=)` | 签名已匹配；返回 `SendResult`（非 dict，引擎不关心返回结构） |
| cron / standing | `HermesCronAPI(platform_name="dingtalk")`，deliver="dingtalk" | `cron_deliver_env_var="DINGTALK_HOME_CHANNEL"`，`standalone_sender_fn=_standalone_send` |

`response_correlation_key` / Tier-1 关联：直接复用 `TagAdapterMixin` 默认实现（基于 `chat_id:message_id`），
无需平台特化（与 Demo/Slack 一致）。

## 3. 配置与注册

### 3.1 base 平台必需项
**重要更正（2026-06-29，已核实）：钉钉标准机器人没有飞书 `im:message.group_msg` 的等价能力。**
钉钉 Stream 客户端只订阅 `dingtalk_stream.ChatbotMessage.TOPIC`（机器人消息主题），该主题**仅推送
「群里 @ 机器人」的消息 + 全部单聊消息**；未 @ 的群消息**根本不会下发到** `_on_message`。因此 base 的
`require_mention` / `free_response_chats` 只是对「已下发消息」的**响应策略**过滤，**无法**让钉钉上游推送
未 @ 群消息。钉钉确有一个全量「用户消息事件」，但仅对**个别钉钉内部企业**白名单开放，非标准开发者可用、
未验证，不能依赖。证据见 §4.4。

**结论：钉钉上 Tier-0 环境上下文（ambient）在标准机器人路径下不可得**——群里只能拿到 @ 机器人的消息。
本插件因此在钉钉上**退化为「仅 @ 的 Tier-1 记忆」**，等同于「未授 `im:message.group_msg` 的飞书」。
这不是配置开关问题，`require_mention` 设成什么都改变不了。I3「无 ambient、仅 @ 回答」自然成立。
`require_mention` 建议保留默认即可（不影响该结论）。

### 3.2 tag 配置块（`extra.dingtalk_tag`，复刻 `slack_tag`）
```yaml
plugins:
  enabled:
    - hermes-tag
platforms:
  dingtalk:
    # require_mention 保留默认即可——它改变不了「群里只能收到 @ 消息」这个上游事实（见 §3.1/§4.4）
    extra:
      dingtalk_tag:
        enabled: true
        enabled_chats:                  # = source.chat_id = 群 conversation_id
          - cidXXXX
        admins:                         # = source.user_id = sender_id（见下方 ID 映射注）
          - <sender_id>
        db_path: dingtalk-tag.sqlite3
        encryption_posture: plaintext-db-on-local-disk
        max_context_chars: 4000
```
`_dingtalk_tag_config()` 复刻 `_slack_tag_config()`：`tier0_context_enabled` 默认 `True`。注意这只是
**插件愿意缓冲**的开关；钉钉群里实际收不到非 @ 消息（§3.1/§4.4），所以群内 Tier-0 通常为空、@ 时才有自身这条。

**ID 映射（已对照 base `_on_message` / `build_source` 确认）：** `source.chat_id` = `conversation_id`（群）
或 `sender_id`（单聊）；`source.user_id` = `sender_id`，`source.user_name` = `sender_nick`，
`source.user_id_alt` = `staff_id`。插件的 `_author` / admin 判定按 `source.user_id` → `user_name` →
`event.author` 取值（base.py:480-481, 673-675），**不读 `user_id_alt`** → 因此 `admins` 必须填 **`sender_id`**
（或 `sender_nick`）；填 `staff_id` 不会命中。`enabled_chats` 填群的 `conversation_id`。

### 3.3 register()（复刻 base dingtalk 的 kwargs，仅换 adapter_factory）
```
name="dingtalk", label="DingTalk Tag",
adapter_factory=adapter_factory,                 # tag 工厂，失败回退 base _build_adapter
check_fn=check_requirements,
is_connected=_is_connected, validate_config=_is_connected,
required_env=["DINGTALK_CLIENT_ID","DINGTALK_CLIENT_SECRET"],
install_hint="pip install 'dingtalk-stream>=0.20' httpx",
setup_fn=<base interactive_setup>,               # 复用 base 的二维码/手填向导
apply_yaml_config_fn=apply_yaml_config,          # 先调 base，再并入 dingtalk_tag
allowed_users_env="DINGTALK_ALLOWED_USERS",      # 必须保留：override 替换整条 PlatformEntry，不继承 base
allow_all_env="DINGTALK_ALLOW_ALL_USERS",        # 同上（base dingtalk register 有这两项，漏写=丢用户白名单）
cron_deliver_env_var="DINGTALK_HOME_CHANNEL",
standalone_sender_fn=<base _standalone_send>,
emoji="🏷️", allow_update_command=True,
# 刻意不设 max_message_length：base dingtalk 的 register 也未设（send() 内部用
# self.MAX_MESSAGE_LENGTH 截断），override 与 base 对齐。
```
> register 是 **last-writer-wins 整条替换**，不是字段级合并 → base dingtalk register 里的每个 kwarg
> （除 adapter_factory/label/emoji 这些有意改的）都要逐项对照复刻，漏写即丢能力（如上面的白名单 env）。
在 `__init__.py:register()` 中追加：feishu 先注册 → slack try/except → **dingtalk try/except**
（注册失败只 warning，绝不拖垮 feishu/slack）。

## 4. 诚实的能力边界（v0.3.0 明确写进 README/文档）

1. **钉钉无 thread、无 reply-target** → `ContextSelector` 只会落到 `deictic_recent` / `plain`，
   `focused_reply` / `thread` 永不触发。证据精度与"无线程平台"一致，属设计事实，非缺陷。
2. **召回类/指示代词类媒体 v1 不支持。** 钉钉 `event.media_urls` 是**远程下载 URL**（由 download_code
   经 robot 文件下载 API 解析而来），非本地文件 → `_persist_event_media`（按本地文件 `exists()/is_file()` 拷贝）
   会跳过 → Tier-0 不缓存图片供后续"那张图"召回。**当前消息的媒体仍直达模型**（base adapter 原生处理远程 URL）。
   延后项见 §7。
3. **主动/定时发送需要有效会话 webhook 或静态 `DINGTALK_WEBHOOK_URL`。** 钉钉回复用每条消息的 session
   webhook（会过期）；对"无近期入站"的群发 enable 通知或 standing-job 会失败，除非配 `DINGTALK_WEBHOOK_URL`
   （走 `_standalone_send`）。standing/cron 文档需标注此项为必需。
4. **【已核实，硬限制】钉钉群里的环境 Tier-0（ambient）不可得 —— 没有飞书 `im:message.group_msg` 的等价物。**
   钉钉标准机器人（含 Stream Mode）**默认且唯一**能在群里收到的是「@ 机器人」的消息，外加全部单聊消息；
   未 @ 的群消息不会下发。证据三方互证：
   - **本仓库依赖的 pinned 适配器**（`plugins/platforms/dingtalk/adapter.py`，Hermes v0.17.0）`connect()` 里
     只 `register_callback_handler(dingtalk_stream.ChatbotMessage.TOPIC, ...)` 一个回调；没有任何「群全量消息」
     主题订阅。`require_mention` / `free_response_chats` 都是对**已下发**消息的响应过滤，不改变上游推送。
   - **钉钉官方/开发者社区**：群机器人只接收 @ 事件；全量「用户消息事件」**仅个别钉钉内部企业白名单可用**，
     非标准开发者可得、未验证；社区原话「无法接收所有未 @ 的群消息——这与飞书的完整事件订阅能力存在差距」。
   - **Hermes 官方 DingTalk 文档**：群里未 @ 的消息不会被机器人接收/处理。

   后果：本插件在钉钉上**退化为「仅 @ 的 Tier-1 长期记忆」**，与「未授 `im:message.group_msg` 的飞书」同档；
   Tier-0 短期环境上下文在群里**结构性缺失**，不是配置或 live-verify 能解决的。这是钉钉与飞书的根本能力差距，
   必须写进 README/AGENTS，避免给出「钉钉=飞书同款环境记忆」的错误预期。

## 5. 测试（stub 模式，复刻 slack 路子，保持 101 测试基线绿）

- 新增 `HERMES_PLUGIN_DINGTALK_USE_STUBS=1` 环境位 + stub `DingTalkAdapter`（仿 slack.py 的 stub 分支）。
- 新增 `tests/test_dingtalk_seam.py`（约 12–15 个用例）：
  - `is_mentioned`：DM / `is_in_at_list` 群 @ / 非 @ 群 / `/tag` 文本快捷判定；
  - `handle_message` 路由：enabled_chats 命中走 engine，未命中走 `super()`；
  - **`dispatch_to_model` 路由（防回归 #1）：** @ 消息经 engine.enhance 后到达 base 的
    `super().handle_message`（即模型分发），**不得**触发 Feishu 的 `_dispatch_inbound_event`；
  - `receive_all` → Tier-0 缓冲非 @ 群消息；仅 @ 触发回答与 Tier-1 写入；
  - 命令处理（`/tag admin count` 等）回执；**且需 @/DM 才处理**（非 @ 群内 `/tag` 无回复，对应 §8）；
  - admin 判定（防回归 #3）：`admins` 填 `sender_id`（落 `source.user_id`）→ 命令成功；非 admin → permission denied；填 `staff_id` 不命中；
  - reply-media seam 为 no-op（`[]` / `("","")`）；
  - `send` 透传签名；
  - `adapter_factory`：`dingtalk_tag.enabled` 关闭或坏配置时回退 base adapter（不 brick）。
- 复用既有平台无关测试：`test_second_channel_fixture.py`（已证明 engine 不依赖 Feishu 子类）。
- 更新 `test_version_agreement`（0.3.0）。
- 运行门：`HERMES_PLUGIN_FEISHU_USE_STUBS=1 HERMES_PLUGIN_SLACK_USE_STUBS=1 HERMES_PLUGIN_DINGTALK_USE_STUBS=1 PYTHONPATH=src python3 -m unittest discover -s tests`。

## 8. Live E2E 验收（目标组织 / 机器，人工门）

1. 钉钉开放平台建应用 + 机器人，开启 **Stream 模式**；`hermes ... setup` 走二维码或手填
   `DINGTALK_CLIENT_ID` / `DINGTALK_CLIENT_SECRET`；如需 standing/cron 再配 `DINGTALK_WEBHOOK_URL`。
2. 配置：启用 hermes-tag；`dingtalk_tag.enabled: true` + `enabled_chats` + `admins`（`require_mention` 默认即可）。
3. 重启网关 → 日志出现 dingtalk connected、`Gateway running with N platform(s)`（feishu/slack 不回归）、
   生成 `dingtalk-tag.sqlite3`（说明 `DingTalkTagAdapter` 生效）。
4. 验收（**MUST PASS**）：
   - @ 门控 + 单聊：群内非 @ 的 `/tag admin count` 无回复；@BOT 后（或单聊）`/tag admin count` 返回
     `tier0=… tier1=… standing_jobs=…`。
   - @ 记忆链路：单聊或 @BOT 连问两次相关问题，第二次能用到第一次的结论（Tier-1 写入并被检索）。
   - **环境上下文（预期为「不可用」，验证退化而非能力）：** 先发非 @ 的"背景：测试项目周五截止"，再 @BOT
     "测试项目什么时候到期？" → **预期回答不含"周五"**（因为非 @ 消息根本没进 Tier-0，§4.4）。这条用来
     确认「钉钉无 ambient」的退化是符合预期的，不是回归。若想验证链路本身，把背景消息也 @BOT 即可命中。

## 9. 一句话总结

钉钉的 base adapter 上游已就绪，0.3.0 = 沿用 Slack 的 mixin 接入：新增一个 `platforms/dingtalk.py`
（override `handle_message` + `dispatch_to_model`、`is_in_at_list` 判 @、reply-media stub、复刻 register/config/
工厂回退）+ 测试 + 文档，Layer A 不动。**关键能力边界（已核实，非风险/非待验证）：钉钉标准机器人在群里只能
收到「@ 机器人」的消息，没有飞书 `im:message.group_msg` 的等价能力 → 钉钉上没有 Tier-0 环境上下文，插件退化为
「仅 @ 的 Tier-1 记忆」。** 这必须诚实写进对外文档，不能宣传成「钉钉=飞书同款环境记忆」。
