## 版本内容(code,2 项)

### F-1 `im:message.group_msg` 启动能力检查 + 显式诊断(S)— ✅ SHIPPED(PR #14,2026-07-09;spec 见 [0.4.0-f1-spec.md](0.4.0-f1-spec.md))

Tier-0 依赖敏感权限 `im:message.group_msg`;未授权时表现为"群里静默不响应"——两个竞品共有的最大支持类痛点,我们能在插件层根治。

- 启动/enable 时探测 Feishu app 是否实际持有该 scope;缺失时:日志 + 给 admin 一条明确诊断(引导到申请页),而不是静默降级。
- 分层:检查的**决策/文案**在 Layer A(`base.py` + `i18n.py`,中英双语);**探测机制**是新 seam 方法(带安全默认值 = "unknown→不打扰",不改既有 seam 签名),Feishu 实现在 `platforms/feishu.py`,Slack/DingTalk 用默认值。
- 验收:无权限的测试 app 启动 → 日志与 admin 提示出现;有权限 → 无噪音;76+ 测试不回归。

### F-2 allowlist/admin 身份归一化到 open_id(S–M)— ✅ SHIPPED(PR #13,2026-07-09;spec 见 [0.4.0-f2-spec.md](0.4.0-f2-spec.md))

审批按钮/事件里 `open_id` vs `user_id` 不匹配是卡片相邻痛点中唯一归我们管的(我们已拥有 allowlist/`admins`/`bot_open_id`)。

**Spike 结论(2026-07-09,盒上实测,只读):**

1. 插件 `_author(event)`(base.py:673)读 `source.user_id`,而该字段语义**依赖 app 的授权 scope**:hermes-agent `plugins/platforms/feishu/adapter.py` `_resolve_sender_profile()`(~L4040)取 `primary_id = user_id or open_id` —— app 持有 `contact:user.employee_id:readonly` 时是租户 user_id,否则回落 open_id。(注意:该文件头部注释声称 user_id 装 open_id,与实现矛盾——文档已过时,以代码为准。)
2. 盒上现状:shiling-pm 的 `granted_scopes` 仅 `im:message.group_msg`;`feishu-tag.sqlite3` 的 tier0_messages 中 DISTINCT author 全部为 `ou_` open_id;配置 `admins: [ou_c382…]`。**今天匹配成立,但仅因 employee-id scope 缺席而侥幸成立**。
3. 潜在断裂:将来任何人给 app 授了 employee-id scope,所有入站事件的 `source.user_id` 整体翻转为租户 user_id → admin gate 静默失效(owner 收到 permission denied),且 tier1 的 owner/asked_by 与历史 `ou_` 行不再同源。
4. 修复路径已验证可行:`MessageEvent.raw_message` 保留原始 lark 事件(`normalized = MessageEvent(..., raw_message=data)`,adapter.py ~L3292),其中 `sender.sender_id.open_id` **恒在**、与 scope 无关。

**设计(据 spike 定稿):**

- 新 seam 方法 `stable_author(event)`,Layer A 声明,**安全默认值 = 现有 `_author(event)`**(Slack/DingTalk 不变);Feishu 实现优先读 `raw_message` 里的 `sender_id.open_id`,缺失时回落 `source.user_id`。不改任何既有 seam 签名。
- Layer A 的 `_is_admin` 判定与 tier1 写入(owner/asked_by)改走 `stable_author`。
- 文档:`admins` 必须填 open_id(`ou_`);标注 `contact:user.employee_id:readonly` 是敏感 scope、本插件不需要它。
- 验收:模拟"授了 employee-id scope"(事件 `source.user_id` 为租户 ID、raw_message 带 open_id)时 admin 命令仍命中;现状(无 scope)行为逐字节不变;76+ 测试不回归。
