# F-2 Implementation Spec — scope-independent sender identity (normalize to open_id)

Status: spec for implementation. 2026-07-09. Author: Claude (architect). Implementer: Codex.
Parent plan: [0.4.0-plan.md](0.4.0-plan.md) §F-2 (spike findings recorded there are the factual basis; do not re-verify).

## Goal

Make every identity consumer in the plugin (admin gate, tier0/tier1 author, context author-match)
see a **scope-independent** Feishu sender ID (the app-scoped `open_id`, `ou_…`), instead of the
current `event.source.user_id` whose meaning silently flips when the Feishu app is granted
`contact:user.employee_id:readonly`.

**Why (settled facts from the spike — trust these):**

- hermes-agent's Feishu adapter fills `source.user_id` with `user_id or open_id` (tenant user_id
  wins when the employee-id scope is granted; open_id otherwise).
- Production config `admins:` uses `ou_…` open_ids; all historical tier0/tier1 rows store `ou_…`.
  Matching works today only because the scope is absent.
- `MessageEvent.raw_message` holds the original lark event; `raw_message.event.sender.sender_id.open_id`
  is always present regardless of scopes. (Real objects are lark SDK objects with attribute access;
  tests may use dicts — support both.)

## Design (decided — do not redesign)

Single ingress normalization hook at the one choke point all platforms funnel through:
`TagEngine.handle_message` (core.py:332). A new seam method with a safe default; only Feishu
overrides it. No per-call-site changes: downstream consumers (`_author` in base.py and
context.py's own `_author`) automatically see the normalized event, so tier0 rows, tier1
owner/asked_by, admin gates, and context author-matching stay mutually consistent by construction.

## Exact changes

### 1. `src/hermes_tag/core.py`

- `PlatformSeam` protocol (line ~291): add one method to the protocol body:
  ```python
  def normalize_inbound_identity(self, event: Any) -> Any: ...
  ```
- `TagEngine.handle_message` (line ~332): AFTER the `enabled_chats` admission check (the
  `admission_dropped` early-return must stay first, so non-enabled chats are zero-touch), BEFORE
  the `receive_all`/`store_tier0` block, insert:
  ```python
  event = self.seam.normalize_inbound_identity(event)
  ```

### 2. `src/hermes_tag/base.py`

- `TagAdapterMixin`: add the default implementation (safe default — identity function):
  ```python
  def normalize_inbound_identity(self, event: MessageEvent) -> MessageEvent:
      return event
  ```
- `preflight_status()` metrics dict (lines ~140-157): add one entry
  `"author_normalized": self.store.metric("author_normalized"),` so the silent-flip scenario is
  observable in prod via `/tag status`.

### 3. `src/hermes_tag/platforms/feishu.py`

- Module-level helper:
  ```python
  def _raw_sender_open_id(event: Any) -> str:
  ```
  Defensive extraction of `raw_message → event → sender → sender_id → open_id`. At every hop:
  try attribute access, fall back to `dict.get` when the node is a dict, return `""` on any
  missing hop. Return the value only if it is a `str` starting with `"ou_"`; else `""`.
- `FeishuTagAdapter.normalize_inbound_identity(self, event)`:
  1. `open_id = _raw_sender_open_id(event)`; if falsy → return event unchanged.
  2. Read current `source.user_id` (same getattr chain as `_author`); if it already equals
     `open_id` → return event unchanged (this is today's prod path — must stay byte-identical,
     no metric increment).
  3. Otherwise rewrite: try `event.source.user_id = open_id` in-place; if that raises
     (frozen dataclass), try `dataclasses.replace(event.source, user_id=open_id)` and assign
     `event.source`; if both fail, return the event unchanged. On successful rewrite call
     `self.store.inc("author_normalized")`.
  4. Return the event.
- Do NOT touch `_dispatch_inbound_event` or any existing seam method.

### 4. Tests — new file `tests/test_identity_normalization.py` (mirror existing test style; unittest, stubs via `HERMES_PLUGIN_FEISHU_USE_STUBS`)

MUST cover, and each test must fail if the corresponding behavior is removed:

1. **Default no-op**: an adapter that doesn't override the hook (Slack/DingTalk path) returns the
   same event object; engine still processes it (existing behavior unchanged).
2. **Non-enabled chat zero-touch**: for a chat not in `enabled_chats`, the hook is never invoked
   (spy/counter on a subclassed hook) and `admission_dropped` still increments.
3. **Scope-flip repair (the F-2 case)**: Feishu event with `source.user_id = "on_tenant_123"`
   (tenant-style, non-`ou_`) and `raw_message` carrying `sender.sender_id.open_id = "ou_admin1"`;
   config `admins: ("ou_admin1",)`; a `/tag admin count` command event must be ACCEPTED (not
   `permission denied`), and a stored tier0 row's `author` must be `"ou_admin1"`; metric
   `author_normalized` == number of normalized events.
4. **No raw open_id**: `raw_message=None` (and separately a raw_message missing `sender_id`) →
   event unchanged, no metric increment, no exception.
5. **Already-normalized (today's prod)**: `source.user_id == raw open_id` → event object identical
   (same `source.user_id`), `author_normalized` metric stays 0.
6. **Dict-shaped raw_message**: hop chain works when `raw_message` is a nested dict.

- `CHANGELOG.md`: under `## [Unreleased]` add a `### Fixed` bullet describing the

## Invariants (MUST NOT violate)

- No changes to any EXISTING seam method signature; `assert_real_seams`/`_assert_signature`
  checks in base.py must keep passing untouched.
- Files that must NOT be edited: `context.py`, `i18n.py`, `adapter.py`, `platforms/slack.py`,
  `platforms/dingtalk.py`, `after-install.md` (root location is load-bearing for the installer),
  `plugin.yaml`, `pyproject.toml` (no version bump in this change).
- context.py stays untouched BY DESIGN: it reads the already-normalized event, so selection
  author-match and tier0 rows remain consistent — do not "helpfully" switch it to a helper.
- Behavior for non-enabled chats and for events whose `source.user_id` already equals the raw
  open_id must be byte-identical (no new sends, no new audit rows, no metric increments).
- English README and README.zh-CN.md must say the same thing.
