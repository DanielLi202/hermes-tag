# Hermes Feishu Tag 插件 — 实现 Spec

> 源自《设计方案 v2.2》。本文是交给 **dev agent** 的实现契约。
> 每个 Spec 只给:**目标** / **关键技术决策(dev 必须遵守的约束与不变量)** / **验收标准(= 验证方式,可观察行为)** / **交给 dev 决定** / **不在范围内**。
> **实现细节(代码结构、具体方法名、SQL、算法、库选择)由 dev agent 自行决定**,只要满足验收标准且不违反关键技术决策。
> 关键词:**MUST=必须**、**SHOULD=应当**。每个 Spec 交付须附最小可运行验证(单测/集成测或可观察行为)。

---

## 全局不变量(适用于所有 Spec)

- **G1** 仅以 Hermes **platform plugin** 形态实现;**不改 Hermes core、不 fork、不 monkeypatch**。
- **G2** `plugin.yaml` **pin 一个 Hermes 版本**;override 经 registry 以 `name=feishu` 重注册生效(last-writer-wins);依赖的 base seam 在启动时做**签名自检**,不符则**响亮失败**(绝不静默降级处理)。
- **G3** 所有数据按 `chat_id` 隔离;**仅 `enabled_chats` 内的群被处理**。
- **G4** 日志**不得**打印 app secret / 用户 token / 完整敏感正文。
- **G5** 永久非目标,任何实现**不得引入**:飞书 docs/wiki/base 检索 grounding、ambient 自发插话、跨群/跨数据源学习。
- **G6** 飞书权限事实:消息读取与资源下载同在 `im:message`(**无独立 `im:resource` scope**);回复字段为 `parent_id`/`root_id`(**无 `upper_message_id`**);全量群消息需敏感 scope `im:message.group_msg`。
- **G7** 本版为**单群试点**(D4):MUST 只接受**恰好一个** `enabled_chats` 的 pilot `chat_id`;配置为零个或多个时**启动失败**。
- **G8** L2/Tier-0 **只能由收到的推送事件喂入**(D1);**禁止**任何历史回扫/history sweep/backfill 作为来源或降级路径。

## Spec 0 — 插件骨架与接入边界(Foundation)

**目标:** 插件以 `feishu` 覆盖内置 adapter 加载;确立"接收 ≠ 存储"边界;建立按群存储、配置、权限自检与可观测。这是其余 Spec 的底座。

**关键技术决策(约束):**
- 继承官方 `FeishuAdapter`,**通过 Hermes platform registry 以 `name=feishu` 重注册覆盖**(D5,last-writer-wins);覆盖发生在 registry/factory,不是 import 时 monkeypatch。(具体注册 API 由 dev 决定。)
- override **最小面**,优先复用基类公开 seam(`handle_message` / `MessageEvent`);依赖到的内部 seam 必须在启动自检清单内(G2)。
- **专用试点 bot/app,仅安装于试点群**;入口 admission 对**非 `enabled_chats` 群的消息事件硬丢弃**(收到即弃,不落库、不处理)(D7)。
- **权限自检**:启动校验实际授予 scope,缺失则关闭对应能力——无 `group_msg` → 关闭 Tier-0 全量摄取与 L2,降级到仅 L1/当前消息;**`@` 交互记忆(Tier-1)不依赖 `group_msg`,仍启用**。
- 按群 **SQLite** store,承载三类数据(Tier-0 / Tier-1 / standing_work);DB 文件权限 **0600**;**声明加密姿态**(是否 SQLCipher / 是否置于加密卷)。
- 配置 schema 见设计 §12;提供可观测计数(见 §"可观测")。

**验收标准(MUST,除非标 SHOULD):**
- A0.1 启动后该 platform 由 `FeishuTagAdapter` 提供,覆盖内置 adapter(可由注册表/启动日志确认)。
- A0.2 故意制造 base seam 签名不符 → 启动**响亮失败并拒绝处理消息**(不静默跑)。
- A0.3 非 `enabled_chats` 群的消息事件**不产生任何落库/处理**(store 无该群行;`admission_dropped` 计数增加)。
- A0.4 在仅授予 `@`-类 scope(无 `group_msg`)下启动:Tier-0/L2 能力关闭、降级到 L1;`@` 交互记忆路径仍启用(自检日志/能力开关体现)。
- A0.5 DB 文件权限为 0600。
- A0.6 **单群试点强制(G7):** `enabled_chats` 恰好一个时正常启动;为 0 个或 ≥2 个时**启动失败**(构造三种配置断言)。
- A0.7 **接收边界证据(D7):** 启动/preflight 输出**证明**:使用专用 Feishu app/bot 身份、恰好一个 pilot 群启用、且明示"`enabled_chats` 是存储/处理边界,非接收边界"。该证据可被 deploy gate / 启动状态读取并断言。
- A0.8 **加密姿态(可断言):** 启动/管理状态暴露 DB 加密姿态(SQLCipher / 加密卷 / 明文)声明;测试断言该声明存在且非空。

**交给 dev:** registry 接入与 `adapter_factory` 写法、SQLite schema/索引、自检实现方式、配置解析库、计数埋点形式。
**不在范围内:** 上下文构建、记忆、standing(后续 Spec)。

## Spec 1 — 闭环一:全量摄取 + Tier-0 消退 + `@` 时 L1/L2 上下文

**目标:** 被 `@` 时,agent 看到 **当前消息 + 引用消息的图片/文件 + 择优的近期未@背景**;未@消息全量摄入 Tier-0 并自由消退,不膨胀。

**关键技术决策(约束):**
- **摄取:** `enabled` 群每条消息(含未@)→ Tier-0;按 `message_id` **幂等去重**(重复投递不重复落库)。
- **消退:** Tier-0 按 age / count **物理删除,自由消退(无 watermark)**;下载的媒体随 Tier-0 TTL 一并清理,不游离窗口外。
- **L1 引用媒体:** 取 `parent_id`/`root_id` → 拉父消息 → 解析 `image_key`/`file_key` → 下载。**两条下载路径**:独立图片消息走 `images/:image_key`;富文本内嵌/文件消息走 `messages/:id/resources/:file_key?type=image|file`。**复用内置下载/cache helper**;字节并入 `media_urls`/`media_types` 交原生 vision。上限 `max_reply_media_items` / `max_reply_media_bytes`。**下载失败 → 降级为文本占位,不中断**。
- **L2 择优背景:** 读 Tier-0,按 **thread/引用链、时间邻近、显式指代("那个截图")、作者标注** 排序**择优**(非"上次回复以来全部消息"照单全收),带作者标签入上下文。
- **预算:** 总预算 `max_context_chars`,优先级 **当前 > 引用媒体 > 择优背景**;不足时从低优先级裁剪。
- **边界:** 未@消息**不触发回复、不进 Hermes 会话、不进模型**——除非在本次真实 `@` 中被当作 L2 拉取。

**验收标准(MUST,除非标 SHOULD):**
- A1.1 被 `@` 且引用了一条**图片**消息 → 该图出现在 `media_urls` 并走原生 vision(集成测试断言增强事件的 `media_urls` 含解析出的图)。
- A1.2 引用**文件**消息 → 文件解析进 media(同上)。
- A1.3 媒体下载失败 → 注入可见**文本占位**,流程不抛错、不中断。
- A1.4 未@消息进入 Tier-0(store 有行);超出保留窗口后被**物理删除**(行消失,`tier0_evicted` 计数增加)。
- A1.5 同一 `message_id` 重复投递**不产生重复行**(幂等)。
- A1.6 L2 只纳入相关消息:构造"Bob 的无关消息 + Alice 铺垫后@"场景,断言 **Bob 无关消息不被无差别注入** Alice 的@上下文。
- A1.7 预算超限时按优先级裁剪,**当前消息与引用媒体不被背景挤掉**。
- A1.8 背景文本**带作者标签**(可断言注入文本含作者归属)。
- A1.9 **未@消息不触发任何 agent 行为(D8 边界):** 构造一条未@消息,断言**无 Hermes runner/模型调用、无 session 创建、无对外回复**,仅有一条 Tier-0 行。
- A1.10 **媒体随 Tier-0 TTL 清理:** Tier-0 某行过期被删时,其下载的媒体缓存(文件/对象)**一并消失**,不游离窗口外。
- A1.11 **禁止历史回扫(G8):** Tier-0 行**只**由推送消息事件创建;无任何 history/backfill 来源(可由代码路径/集成测试确认无历史拉取调用)。

**交给 dev:** 择优打分算法、SQL/索引、媒体缓存路径与命名、预算裁剪具体策略、拉取父消息的调用实现。
**不在范围内:** 长期记忆、standing work。

## Spec 2 — 闭环二:`@` 交互长期记忆 + 注入 + 任务级 session

**目标:** 每次 `@` 处理后,把交互产物写入按群长期记忆;后续 `@` 能回忆;执行用任务级 session,避免并发@互相污染。

**关键技术决策(约束):**
- **Tier-1 只从 `@` 交互写入**:每次 `@` 处理后写一条记忆,含 **本次问题 / 感知上下文要点 / agent 结论或决策 / owner**;带 **provenance**(`trigger_message_id`, `asked_by`, `source_message_ids`/时间区间, `status`, `confidence`)。**不**对全量流后台蒸馏(Tier-0/Tier-1 解耦)。
- **注入:** `@` 时把相关 Tier-1 记忆按预算注入(优先级低于背景)。
- **consolidation:** Tier-1 设上限,旧记忆衰减/合并,但合并保留**可追溯来源/归属**。
- **会话归属:** 按 **thread/请求(任务)级 session**,每次注入群记忆;**不共享 live session**(避免并发@污染);真正群级 session 仅在确认 Hermes 并发安全后才考虑。
- **删除联动:** 用户删除/某群禁用能清理派生记忆(deletion linkage,配合 Spec 4)。

**验收标准(MUST,除非标 SHOULD):**
- A2.1 一次 `@` 得出结论后,**后续相关 `@` 的注入上下文含该结论/owner**(集成测试:第二次@的上下文包含第一次的记忆条目)。
- A2.2 **隐私不变量(确定性场景):** 构造"未@无关消息 + 被 L2 选中的消息 + 一次 `@` 交互"的固定场景,断言写入的 Tier-1 记忆 **provenance 只指向该 `@` 交互及其选中的来源上下文**;**不含未被@引用的群消息**。
- A2.3 两个用户**并发 `@`**,各自任务状态**不互相污染**(构造并发场景,断言任务级 session 隔离)。
- A2.4 禁用某群 → 该群 Tier-1 记忆被清除(配合 Spec 4)。
- A2.5 **删除联动:** 删除来源/触发消息,或走支持的删除路径,**按 provenance key 移除或 tombstone 派生的 Tier-1 记忆**(构造场景断言派生记忆不再被检索/注入)。
- A2.6 **consolidation 上限:** 持续写入超过上限后,Tier-1 条数受上限约束(consolidation 触发),且合并后**来源/归属仍可追溯**。
- A2.7 **完整预算优先级:** 构造 Tier-1 + L2 合计超预算场景,断言**当前消息与引用媒体存活、Tier-1 最先被裁剪**(优先级 当前>引用媒体>背景>长期记忆)。

**交给 dev:** 记忆条目的生成/摘要方式、注入选择算法、session key 实现、consolidation 策略。
**不在范围内:** Tier-0 摄取(Spec 1)、standing(Spec 3)。

## Spec 3 — 闭环三:Standing work(定时任务)

**目标:** 用户**显式**交代的定时任务映射成 Hermes cron;可列出/取消;定时外发受控且可审计。

**关键技术决策(约束):**
- **创建需显式命令 + bot 回确认后才生效**(避免闲聊误建,如"每周五总结会不会太晚")。
- **稳定 job ID**;列出展示 ID;**取消按 ID 或确认过的精确匹配**(避免误删相似/他人任务)。
- 映射 **Hermes cron**,scoped `chat_id`;registry 表:`(chat_id, description, schedule, timezone, cron_job_id, owner, status, created_at)`。
- **权限:** 仅 owner/admin 可建/取消。
- **时区:** 显式来源与展示,处理 DST。
- **定时外发 = 唯一授权的非@开口**:有可见 registry、可暂停/禁用、可审计;**非** bot 自发插话(不得引入 ambient,G5)。

**验收标准(MUST,除非标 SHOULD):**
- A3.1 普通闲聊**不创建**任务(构造"每周五总结会不会太晚" → 无 job 创建)。
- A3.2 显式命令 → bot **先回确认** → 确认后才注册 cron job(确认前 store 无 job)。
- A3.3 列出返回**带稳定 ID** 的任务;**按 ID 取消只删目标 job**,不误删相似任务。
- A3.4 到点 cron 触发**回贴该群**(scoped `chat_id`),且该外发可在 registry/审计中看到。
- A3.5 非 owner/admin 的建/取消被**拒绝**。
- A3.6 **时区/DST 正确(确定性 fixture):** 给定时区的"每周五10:00"按该时区触发;跨 DST 边界仍正确(固定 fixture 断言触发时刻)。
- A3.7 **暂停/禁用:** 一个 standing job 可被**暂停/禁用**后**停止触发**,registry 状态体现;可重新启用或移除。

**交给 dev:** 意图解析与确认交互的实现、cron 注册 API 调用、ID 生成方式。
**不在范围内:** ambient 主动行为(永久非目标)。

## Spec 4 — 隐私生命周期与管理命令(横切)

**目标:** 满足诚实披露、保留、审计、禁用即删、管理控制。

**关键技术决策(约束):**
- **启用告知:** 入群/启用增强时,向群一次性告知,措辞与 **D8 一致**:消息将被记录并短期缓冲、被@时相关消息可能作背景进模型;**长期记忆仅来自 `@` 交互,不含未被引用的群消息**。
- **保留表:** Tier-0 / Tier-1 / 媒体 / cron 各自的保留期与删除规则成文成表。
- **审计事件:** 启用/关闭/清空/建/取消任务/导出 等留审计。
- **禁用即删:** 对某群关闭增强 → **级联删除** Tier-0/Tier-1/媒体/该群 cron。
- **管理命令(限 owner/admin):** 清空某群上下文、查看缓存条数、关闭某群增强。
- 日志遵守 G4;加密姿态遵守 Spec 0。

**验收标准(MUST):**
- A4.1 启用某群触发**实际面向群的一次性告知消息**(断言群内确有该告知发出;审计记录可作附加证据,但**不能**以审计代替告知)。
- A4.2 禁用某群后,**级联清除证据三件**:DB 中 Tier-0/Tier-1 行消失、媒体缓存文件/对象消失、**该群 Hermes cron job 已注销/不再触发**(分别断言;cron 注册与媒体缓存可能在 DB 之外)。
- A4.3 管理命令**仅 owner/admin** 可用;清空/计数命令行为正确(计数与实际行数一致)。
- A4.4 审计事件被记录(启停/清空/建取消任务可在审计日志查到)。
- A4.5 日志中**不出现** app secret/token/完整敏感正文(抽查)。
- A4.6 **告知内容(内容级断言):** 告知文案覆盖三点——(a) 所有群消息(含从未与 bot 交互的成员)被本地记录并短期缓冲;(b) 相关消息仅在被 `@` 时可能进入模型;(c) 长期记忆仅来自 `@` 交互。逐点断言。
- A4.7 **保留表存在:** Tier-0/Tier-1/媒体/cron 四类的保留期与删除规则以**表/管理输出/文档工件**形式存在且四类齐全(断言该工件存在并含四类)。

**交给 dev:** 告知文案投递方式、审计存储形式、命令实现、加密(若启用)接法。

## 可观测(横切,设计 §15)

至少暴露计数/状态:`admission_dropped`、Tier-0 行数 / `tier0_evicted`、Tier-1 记忆条数 / 写入失败、媒体下载成功/失败、降级触发(无 `group_msg`)、override 自检结果、standing job 数。

**验收(MUST):** AO.1 上述每个计数/状态**可查询**,且能由一个 fixture 触发其更新(构造对应事件后断言计数变化 / 状态可读)。

## 交付与验证总纲

- 每条 **MUST** 必须有可运行验证(单测/集成测或明确可观察行为);dev 交付每个 Spec 时附最小验证集。
- 闭环顺序:Spec 0 → 1 → 2 → 3,Spec 4 横切(随 0/1/2/3 增量满足)。每闭环**独立可用、独立验证**。
- 评审(Codex/Claude)按各 Spec 的 MUST **逐条核**;未满足即不通过。
